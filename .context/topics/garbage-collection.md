# 垃圾回收（Garbage Collection）

## 核心理念

> 默认熵增：AI 高速产出必然带来技术债、文档漂移、重复实现。必须设计专门的"垃圾回收"系统性对抗熵增。

## 架构级技术债扫描（存量治理）

"架构级"意味着不是 lint 能查到的格式问题，而是跨文件/跨模块的结构性问题。

### 分类框架（Fowler 技术债四象限）

参考 Martin Fowler 的技术债象限模型，按**意图（故意/无意）** × **性质（谨慎/鲁莽）** 分类：

| | 谨慎 (Prudent) | 鲁莽 (Reckless) |
|--|---------------|----------------|
| **故意 (Deliberate)** | 明知有债但为赶 deadline，有计划偿还 | 明知好设计是什么但选择快速实现，低估利息 |
| **无意 (Inadvertent)** | 团队成长后发现了更好的设计方案，债是学习成本 | 不了解好设计，产出混乱代码 |

扫描时应识别每笔债的象限归属，不同象限处理策略不同：
- **谨慎故意**：设 repayment plan，按计划还
- **鲁莽故意**：立即拦截（Hook L2守护）
- **谨慎无意**：不可避免，拥抱重构
- **鲁莽无意**：培训 + 规范 + Hook 教育

### 可复用做法

- 技术债扫描设为周期性任务（如每次大 feature 合并后）
- 扫描输出结构化报告
- 严重问题自动生成修复计划（与 Hook 的"增量拦截"互补）

## 文档新鲜度巡检（防 Doc Drift）

> 过时文档比没有文档更危险

### 两个维度

- **机械维度**：路径、命令、引用是否失效（可脚本化）
- **语义维度**：描述是否过时（需结合代码语义判断）

### 第一道防线

把"文档更新"写进 ExecPlan 的验收标准。巡检只是兜底。

## 执行策略

1. 每个 ExecPlan 的验收标准中，包含"更新相关文档"条目
2. 周期巡检定期扫描全仓库文档健康度
3. 扫描结果进入优化飞轮（见 `optimization-loop.md`）
4. 大版本/大 feature 合并后触发技术债扫描
