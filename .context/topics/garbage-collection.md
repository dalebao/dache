# 垃圾回收（Garbage Collection）

## 核心理念

> 默认熵增：AI 高速产出必然带来技术债、文档漂移、重复实现。必须设计专门的"垃圾回收"系统性对抗熵增。

## 架构级技术债扫描（存量治理）

"架构级"意味着不是 lint 能查到的格式问题，而是跨文件/跨模块的结构性问题。

### 典型扫描规则

| 规则 | 说明 |
|------|------|
| 重复实现 | 同一逻辑在多个模块独立实现 |
| 数据模型跨层分裂 | 同一实体在 API/DB/UI 层各自定义 |
| 配置散落 | 同类配置分布在多个文件中 |
| 相似流程未抽象 | 多个流程有相同骨架但各自独立实现 |
| 重复持久化无一致性 | 同一数据写多份但无事务保证 |

### 可复用做法

- 技术债扫描设为周期性任务（如每次大 feature 合并后）
- 扫描输出结构化报告
- 严重问题自动生成修复计划（与 Hook 的"增量拦截"互补）

## 文档新鲜度巡检（防 Doc Drift）

> 过时文档比没有文档更危险

### 两个维度

- **机械维度**：路径、命令、引用是否失效（可脚本化）
- **语义维度**：描述是否过时（需结合代码语义判断）

### 第一道防线

把"文档更新"写进 ExecPlan 的验收标准。巡检只是兜底。

## 执行策略

1. 每个 ExecPlan 的验收标准中，包含"更新相关文档"条目
2. 周期巡检定期扫描全仓库文档健康度
3. 扫描结果进入优化飞轮（见 `optimization-loop.md`）
4. 大版本/大 feature 合并后触发技术债扫描
