# 优化闭环（Optimization Loop）

## 核心理念

> 没有分类，只有故事；有了分类，故事变趋势，趋势变优先级。

## 信号来源

三条信号渠道，持续收集 Harness 运行状态：

| 来源 | 内容 | 采集方式 |
|------|------|----------|
| 主观反馈 | 研发/使用者的感受与抱怨 | 定期 poll / 复盘 |
| 业务信号 | CI 失败、MR 退回、revert、线上事故 | 自动化采集 |
| Agent Trace | Agent 执行轨迹中的卡点/重试/失败 | 日志分析 |

## Problem Taxonomy（问题分类）

所有信号用稳定标签体系归类：

| 类别 | 含义 |
|------|------|
| Navigation Gap | 执行者找不到正确的文档/信息 |
| Doc Drift | 文档已过时，与代码/事实不一致 |
| Verification Ambiguity | 验收标准模糊，无法判定成功/失败 |
| Guardrail Hit | Hook/门禁正确拦截（正常行为） |
| Guardrail Noise | Hook/门禁误报，不该拦的拦了 |
| Env Drift | 环境差异导致的问题 |

## 改进飞轮

```
Taxonomy 标注             ← 每条信号打标签
    ↓
归堆找高频痛点           ← 周期性聚合，找 Top N
    ↓
定向改进                 ← 补文档/改 Hook/加规则/调边界
    ↓
观测同类问题频率         ← 改进后同一标签是否下降
    ↓
周期性 AB 评测           ← 对照实验校准长期趋势
    ↓
（回到 Taxonomy 标注）
```

## 落地要点

1. Taxonomy 标签必须稳定，不能每个周期换一套
2. 归堆用频率而非严重度（高频小痛 > 低频大痛）
3. 改进后至少观测一个完整迭代周期再下结论
4. AB 评测用于校准"没有 Harness 会怎样"的因果假设
