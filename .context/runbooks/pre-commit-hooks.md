# Pre-commit Hook 操作手册 — L3 稳定事实

## 三层防线定义

### L0 — 通用卫生

对所有项目通用的检查，不依赖具体语言：

- 文件格式（trailing whitespace、换行符、BOM）
- 大文件拦截
- 合并冲突标记检查
- 密钥/凭证泄露扫描
- 分支保护规则

### L1 — 语言质量

按技术栈定制的检查：

- **Go**：`gofmt`、`go vet`、`go mod tidy`
- **Java**：编译检查、Checkstyle、PMD、SpotBugs
- 类型检查、lint

### L2 — AI 专项守护

AI 生成代码特有的检查：

- 占位符拦截：`TODO`、`FIXME`、`HACK`、`XXX` 未经确认不可提交
- 幻觉标记拦截：虚假的文件路径、API 引用、版本号
- 文件行数上限：防止 AI 产出超长巨型文件
- ExecPlan 完成度守护：引用到 ExecPlan 的变更必须满足验收标准

## 报错即指导原则

错误信息必须包含：

```
[拦截类型]：[简短说明]
  路径：相对路径
  当前值：[具体值]
  阈值：[期望值/限制]
  修复建议：[具体命令或操作步骤]
  参考：指向相关文档的相对路径
```

## 增量校验策略

如果仓库已有大量存量问题：
1. 扫描 baseline，记录当前状态
2. 只拦截 diff 中新增的问题
3. baseline 问题进入技术债扫描（见 `garbage-collection.md`）

## 自愈闭环流程

```
提交触发 Hook → Hook 报错（含修复指导）
    → Agent 读取报错 → 自动修复
    → 重新提交 → Hook 通过 → 进入 CR
```

人类角色：定方向 + Code Review，不处理机械化修复。
